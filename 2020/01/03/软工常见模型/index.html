软工常见模型

发表于2020-01-03更新于2021-06-27

瀑布模型

  • 特点
    • 阶段性的顺序性和依赖性
      • 各项活动严格按照线性方式进行，当前活动接受上一项活动的工作结果，完成所需要的工作内容。
    • 推迟实现
      • 在软件设计阶段只考虑系统的逻辑模型，等到编码阶段再来完成程序清单
    • 保证质量 - 每一阶段都必须完成规定的文档并对文档进行复审
  • 优点
    • 可迫使开发人员采用规范的方法
    • 严格地规定了每个阶段必须提交的文档
    • 要求每个阶段交出的所有产品都必须经过质量保证小组的仔细验证
  • 缺点
    • 在可运行的软件产品交付给用户之前，用户只能通过文档来了解产品是什么样的
    • 要求用户不经过实践就提出完整准确的需求，在许多情况下都是不切实际的
    • 瀑布模型几乎完全依赖与书面的规格说明，很有可能导致最终开发出来的软件无法真正满足用户的需要
  • 适用范围
    • 适用于需求明了且时间要求宽松的软件开发项目或者规模小，需求简单，功能单一的项目

快速原型模型

  • 特点
    • 快速建立起在计算机上可以运行的程序，它所完成的功能往往是最终产品的一个子集，其主要目的是确定客户需求
  • 优点
    • 克服了瀑布模型的缺点，减少软件需求不明确带来的开发风险
    • 软件产品的开发基本上是按线性顺序进行的
    • 有助于保证用户的真实需要得到满足
  • 缺点
    • 快速建立起来的系统结构加上连续的修改可能导致软件质量低下
    • 所用的开发技术和工具不一定符合主流
  • 适用范围
    • 适用于客户能提出一般性目标，但不能详细描述输入输出需求的项目

增量模型

  • 特点
    • 融合了瀑布模型的基本成分和原型模型的迭代特征，采用随着日程时间进展而交错的线性序列，把软件产品作为一系列增量来构建设计
  • 优点
    • 能在较短时间内向用户提交可完成的一些有用的产品
    • 逐步增加产品功能可以使用户有较充裕的时间学习和适应新产品，从而减少一个全新软件可能给客户组带来的冲击
    • 开发人员分配灵活
  • 缺点
    • 并行开发构件有可能遇到不能集成的风险，软件必须具备开放式的体系结构
  • 适用范围
    • 适用于已有产品的升级或新版本的开发
    • 完成期限有严格要求
    • 对开发领域比较熟悉或者有原型系统

螺旋模型

  • 特点
    • 将瀑布模型和原型模型结合起来，是典型的迭代模型，并加入两种模型都忽略的风险分析，其特点是风险驱动，关注风险
  • 优点
    • 灵活性，可以在项目的各个阶段进行变更
    • 客户始终参与每个阶段的开发，保证项目不偏离正确方向
    • 对可选方案和约束条件的强调有利于已有软件的重用
    • 也有助于把软件质量作为软件开发的一个重要目标
    • 减少了过多测试或测试不足所带来的风险
  • 缺点
    • 需要有相当丰富的风险评估经验和专业知识，否则风险处理不当会造成损失
  • 适用范围
    • 适用于大规模软件项目，需求不明朗和风险比较高的项目
